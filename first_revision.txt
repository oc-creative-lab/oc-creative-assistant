## II. Target Product Form
Launch the app
↓
HomeLanding (home page, only two large buttons)
├─ [💬 Chat] → ChatEntry (select existing project / create new project) → ChatWorkspace
└─ [📚 Project Library] → ProjectLibrary (grid of project cards) → Workspace
ChatWorkspace features:

Chat Agent A proactively raises questions (guided in reverse by the Question Planner)
The user freely shares ideas
Structured Agent B asynchronously extracts entities/relations/fields in the background
The extracted content → AgentStagingORM (staging) → reviewed and committed in the workbench
The user can exit at any time → return to the home page

Workspace (the workbench, /workspace/:projectId/...):
Left panel (WorkspaceSidebar):
- Project name / description
- Storyline      ──→ PlotCanvas (Vue Flow, renders only PlotNode)
- Character cards ──→ CharacterCardList (Notion-style card grid)
└─ /:charId → CharacterCardDetail (free-form field editing)
- Worldbuilding   ──→ WorldCanvas (Vue Flow, renders only WorldNode)
Associations between the three content types are established automatically (cross-sub-graph edges)
An agent can be summoned for assistance (FloatingChatDock is retained in the workbench)

## III. Key Architectural Decisions (Must Be Strictly Followed)

### Decision 1: Three Independent Sub-Graphs per Project
ProjectORM:
id, name, description
plot_graph_id     (FK → GraphORM)
character_graph_id (FK → GraphORM)
world_graph_id    (FK → GraphORM)
GraphORM (new):
id, project_id, section enum('plot'|'character'|'world')
NodeORM:
graph_id (FK, strengthened) — a node belongs to a specific sub-graph

### Decision 2: Character Cards Reuse CharacterNode Data, Rendered in a New View

- CharacterGraph is still a graph at the data layer (it has nodes + edges)
- The frontend **does not use Vue Flow**; it uses CharacterCardList instead (a Notion-style grid)
- A Character card's "free-form fields" go into the CharacterNode's metadata JSON
- Relationship edges between characters (mentor/enemy/faction) are visualized as tags on the cards; **no lines are drawn**

### Decision 3: A Separate ProjectSeedORM Table
ProjectSeedORM (new):
id, project_id (FK), version (int),
seed_json (text),
source enum('chat_end' | 'user_edit'),
created_at

### Decision 4: Chat Agent Startup Injection = Seed + RAG + Last N Turns

- Do not feed the full conversation history (uncontrollable cost)
- At startup, assemble: Project Seed (~500 tokens) + RAG top-3 (~1k tokens) + last 10 turns of short-term memory (~1k tokens)
- Keep the total context within 2.5–3k tokens

### Decision 5: Two LLM Agents Collaborating

- **Chat Agent A** (facing the user): streaming replies + naturally raising questions
- **Structured Agent B** (background):
  - `structured_extractor`: extracts entities/relations/field suggestions → AgentStagingORM
  - `question_planner`: guides A in reverse on what to ask next (injected into A's prompt)
- A and B **run in parallel** within LangGraph; B's results do not block A's reply

### Decision 6: Introduce Pinia (Replacing Cross-Route composable Sharing)

- Do not delete the existing composables; they continue to handle fine-grained operations within a single canvas
- New Pinia stores: `useProjectStore` / `useGraphStore` / `useChatStore` / `useLibraryStore`
