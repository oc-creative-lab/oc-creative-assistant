# Project Specification — OC Creative Assistant System (Group 8, JC3510)
Source of truth: the project proposal we have already had approved. Every item below is to be treated as a [Fixed Requirement].
Do not deviate from the architecture, modules, design patterns, or constraints specified here without my explicit permission.

## 0. Core Proposition (Non-Negotiable)
This system [assists] the creative process; it [never generates] narrative content.
All AI output is guidance, questions, summaries, or structured suggestions—it never generates story or character prose.

## 1. Product
An Electron desktop PoC: a node-based creative workbench for OC (Original Character) creators.
Users organize fragmentary ideas as nodes; AI agents provide assistance within strict boundaries; RAG anchors
suggestions to the lore the user has imported themselves. Desktop-first—all user creative data is stored
on the local device.

## 2. Architecture (Three Layers—Not to Be Refactored)
- Interface Layer: Electron + Vue + Vue Flow. Contains the node editor,
  visualization panels, and the agent sidebar. Communicates with the backend via REST API.
- Service Layer: FastAPI backend. Agent orchestration via LangGraph,
  organized as a StateGraph with [conditional edge routing] (rather than linear calls). Uses LangChain.
- Data Layer: SQLite stores structured project data; ChromaDB maintains [exactly three]
  vector collections—worldbuilding, characters, plot; a pluggable LLM API layer
  supports switching between OpenAI / DeepSeek.

## 3. Core Modules (Exactly These Four)
1. Node-based Editor — a Vue Flow-based workbench. Nodes = creative fragments
   (Character / Worldbuilding / Plot). Edges = logical relationships. Supports non-linear, fragmentary ideation.
2. Constrained Autonomous AI Agents — [exactly three], each with capabilities
   limited to creative assistance only:
   - Inspiration Agent: asks questions / gives prompts to expand ideas.
   - Research Agent: structured knowledge retrieval.
   - Structure Agent: reorganizes nodes into Character cards / relationship graphs / Plot frameworks.
   All output = guidance / summaries / structured suggestions. Never generates narrative prose.
3. RAG-based Memory Module — user-imported lore is stored in the three ChromaDB
   collections above; retrieved via basic similarity search; injected into the LangGraph shared state when an agent is invoked. The index is incrementally rebuilt
   when node content changes.
4. Structural Visualization — relationship graphs / plot structure views derived directly from existing node + edge data.
   Reads node/edge data directly from SQLite. Contains no independent business logic.

## 4. The "Assist, Not Generate" Constraint — Enforced at [All Three Layers]
1. The system prompt restricts the output format.
2. .with_structured_output() enforces a predefined JSON schema on every agent response.
3. The UI only renders [structured cards] (it never renders freely generated prose).

## 5. Design Patterns (Mandatory—Implemented Strictly at the Specified Locations)
- Component-Based Design: each node type is an independent Vue component.
- Factory Pattern: node instantiation.
- Open/Closed Principle: adding new node types [does not modify] existing logic.
- Strategy Pattern: (a) all agents share a unified interface, dynamically routed by LangGraph;
  (b) switching the LLM model at the API layer (OpenAI / DeepSeek).
- Observer Pattern: node content changes → incremental rebuild of the ChromaDB index;
  visualization redraws on state changes.

## 6. Technology Stack (Fixed)
- Frontend: Vue + TypeScript, Vue Flow, Electron
- Backend: Python, FastAPI, LangGraph, LangChain
- Data: SQLite, ChromaDB, OpenAI / DeepSeek API
- Toolchain: Electron Builder, Pandoc
- Export targets: Markdown + PDF

## 7. Architecture-Level Risk Mitigation (To Be Built Into the System)
- LLM API unavailable: hot-swappable multiple models via the Strategy Pattern + exponential backoff.
- ChromaDB index corruption: transactional writes + integrity check at startup + automatic rebuild from SQLite.
- Node–index desync: hash-based consistency validation + forced index rebuild.
- LangGraph state bloat: a 2000-token cap on RAG + summary compression.

## 8. Explicitly [Out of Scope] (Do Not Build)
- Advanced RAG: no multi-index structure, no semantic routing, no long-term memory.
- No automated workflow execution (no automatic node triggering / pipeline execution).
- No collaborative creation (no multi-user, no version control, no conflict resolution).
